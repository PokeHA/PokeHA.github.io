10.1 异常概述

Java 的异常处理机制可以将“业务代码”和“异常处理代码”分离，提高代码可读性。

  1. 异常处理机制的关键字
    • try：用于包裹可能会发生异常的代码块。
    • catch：捕获并处理指定类型的异常。
    • finally：无论是否抛出异常，都会执行，用于回收资源。
    • throws：声明方法可能抛出的异常。
    • throw：实际抛出一个异常。
  2. 异常的分类
    • Checked 异常：
      • 编译时必须处理的异常，所有的 Checked 异常都必须明确处理（捕获或声明）。
    • Runtime 异常：
      • 运行时异常，可以选择不处理，常见如 NullPointerException。

10.2 异常处理机制

  1. 使用try-catch捕获异常
    • 当try块中的代码在运行时出现异常时，系统会自动生成一个异常对象，并将其交给 JVM。JVM 会查找匹配的 catch 块，找到后进行处理，若找不到，程序会退出。

《图解HTTP》笔记 第一章 了解Web及网络基础

1.1 使用HTTP协议访问Web

当我们在浏览器中输入 URL 并返回显示内容时，实际上使用的就是 HTTP（HyperText Transfer Protocol，超文本传输协议）来进行数据传输。

1.2 HTTP的诞生

  1. 为知识共享而规划 Web
    • 基本理念：通过超文本将互联网连成可互相参阅的万维网（WWW）。
    • WWW 的三项技术：
      1. HTML：文本标记语言。
      2. HTTP：文档传输协议。
      3. URL：指定文档所在地址。
    • WWW 简称为 Web。
  2. Web 的成长时代
    • 各种 Web 客户端和服务器陆续出现，推动了 Web 的发展。
  3. HTTP 的发展历程
    • HTTP/0.9：初始版本，未正式公布为标准。
    • HTTP/1.0：正式标准，仍然在许多服务端广泛使用。
    • HTTP/1.1：目前最主流的版本。
    • HTTP/2.0：新版本，尚未完全普及。

1.3 网络基础 TCP/IP

  1. TCP/IP 协议族
    • TCP/IP 是一系列网络协议的集合，负责管理网络通信。
  2. TCP/IP 的分层管理
    • TCP/IP 协议分为四层：
      1. 网络接口层：负责相邻节点之间的数据传输。
      2. 网际层：实现主机间的逻辑通信，负责选择传输路径（包括 IP 协议）。
      3. 传输层：实现主机进程间的逻辑通信（包括 TCP、UDP）。
      4. 应用层：用户与网络的接口，应用层协议如 HTTP、DNS、FTP 等。
    • 分层设计的好处：
      1. 每层只需要解决其特定的问题，简化了设计。
      2. 分层后，各层之间的接口定义明确，改变某层的内部实现不会影响其他层。

《图解HTTP》笔记 第三章 HTTP报文结构

  1. HTTP报文
    • HTTP报文 是 HTTP 协议在通信时传递的信息单元。
      • 请求报文：客户端发送的报文。
      • 响应报文：服务器端返回的报文。
    • HTTP 报文由 报文首部 和 报文主体 组成。
  2. 请求报文和响应报文的结构
    • HTTP 报文首部包含不同类型的首部字段。
    • 首部字段 分为四种类型：
      1. 请求首部：客户端发送的请求报文包含的首部信息。
      2. 响应首部：服务器发送的响应报文包含的首部信息。
      3. 通用首部：请求和响应报文都可以使用的通用信息。
      4. 实体首部：用于描述报文主体内容的信息。
    • 这些首部字段的具体作用会在后面的章节中详细解释。
  3. 数据压缩
    • 简介：通过对报文内容进行压缩，减少数据体积，提高传输效率。
    • 报文主体和实体主体：二者指代相同的内容，但未压缩时称为报文主体，压缩后称为实体主体。
    • 内容编码（Content-Encoding）：用于指定报文压缩的算法。常见的编码方式有：
      1. gzip：常见的压缩算法。
      2. compress：另一种压缩算法。
      3. deflate：压缩方式。
      4. identity：不进行压缩。
    • 分块传输编码（Transfer-Encoding）：
      • 报文通过 Transfer-Encoding 分块传输，便于处理大文件或未知长度的文件传输。

《图解HTTP》笔记 第二章 简单HTTP协议

  1. Client/Server架构
    • HTTP 协议采用 客户端/服务端 架构。
      1. 客户端 发送请求。
      2. 服务端 返回响应。
  2. 请求和响应
    • 通信流程：客户端通过 HTTP 发送请求报文，服务端接收到请求后返回响应报文。
    • 无请求无响应：如果服务端未收到请求，不会主动发送响应。
  3. HTTP 不保存状态
    • HTTP 是无状态协议，不会保存先前的请求或响应信息。
    • Cookie：通过使用 Cookie，可以在 HTTP 中保存和管理状态，实现客户端与服务端之间的状态记录。
  4. 请求 URI 定位资源
    • HTTP 使用 URL 来定位和访问资源。
  5. HTTP 方法
    • HTTP 使用 请求方法 来指定对资源的操作，常用的请求方法有：
      1. GET：获取资源。
        • 用于请求指定 URL 的资源。
      2. POST：传输请求体。
        • 向服务器提交数据，数据放在请求体中。
      3. PUT：传输文件。
        • 用于上传文件，文件数据放在请求体中。
      4. HEAD：获取报文首部。
        • 和 GET 类似，但只返回响应头部，不包含响应体。
        • 通常用于检查资源是否有效。

《MySQL是怎样运行的》笔记 第九章 InnoDB表空间详解

简介

表空间是一个抽象的概念，对应文件系统中一个或多个实际文件。每个独立表空间都对应一个 .ibd 文件。表空间由页组成，插入记录时是从表空间中拿出一个页来写入。本章会讲解表空间的细节。

9.1 回忆一些旧知识

  1. 页面类型：
    • InnoDB 以页为单位管理存储空间。
    • 数据页的类型名为 FIL_PAGE_INDEX。
  2. 页面通用部分：
    • 所有页面都有两个通用部分：File Header 和 File Trailer。
    • File Header 记录页面相关信息，如页号、页类型、上一页、下一页、校验和等。
    • File Trailer 主要用于校验页面的完整性，防止断电导致页面更新不完整。
    • File Header的部分信息：
      1. FIL_PAGE_OFFSET：即页号，有4个字节，最多支持2^32个页，意味着表空间最多可支持64TB的数据。
      2. FIL_PAGE_PREV 和 FIL_PAGE_NEXT：存储前后页号，形成双向链表，主要用于 INDEX 类型的页面。
      3. FIL_PAGE_TYPE：页面类型。

9.2 独立表空间结构

本节主要讲解系统表空间和独立表空间的结构。系统表空间较为复杂，因此先从系统表空间开始。

  1. 区 (Extent) 的概念：
    • 简介：
      1. 为了更好地管理页，引入了区的概念。
      2. 每连续64个页组成一个区，一个区大小为1MB。
      3. 每256个区组成一组。

  1. 索引虽然快，但在时间和空间上都有代价：
    • 空间代价：
      1. B+树的每个节点都是一个数据页，一个页默认占用16KB，因此会有不少开销。

      1. 对数据进行增删改操作后，需要对每个索引进行维护。
      2. 建立的索引越多，维护的开销也越大。

  1. 简介：
    • B+树并非万能，并不是所有的查询都可以走索引。

《MySQL是怎样运行的》笔记 第六章 B+树索引

  1. 没有索引的查找
    • 在没有索引的情况下，查找记录的操作如下：
    1. 在一个页中查找
      • 假设记录数量较少，全部存储在一个数据页中。
      • 有两种情况：
        1. 以主键为搜索条件：
          • 先对页目录使用二分法找到对应的槽，然后遍历得到目标记录。
        2. 以其他字段为搜索条件：
          • 只能遍历所有记录进行查找。
    2. 在多个页中查找
      • 如果表中的记录存储在多个数据页中，查找分为两个步骤：
        1. 使用双向链表遍历每个数据页，因为页之间不是有序的。
        2. 在每个数据页中对记录进行查找。
      • 问题：
        • 没有索引时，需要遍历所有数据页才能找到目标记录。
  2. 使用索引的情况
    1. 一个简单的索引方案
      • 通过以下步骤优化查找效率：
        1. 让所有数据页按主键有序排列。
        2. 为每个数据页建立目录，目录项存储数据页中最小记录的主键和页号。
        3. 通过二分法在目录中查找目标记录所在页，然后在该页中查找具体记录。
        4. 该目录即为索引。
